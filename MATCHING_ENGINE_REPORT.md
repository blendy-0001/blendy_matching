## 全体アーキテクチャ

### システム構成図

```
入力: メンバーリスト（Notion DB）
        ↓
Step 0: マッチング済みペアの抽出（再マッチング防止）
        ↓
Step 1: 全ペア生成（combinations）& Activities データ取得
        ↓
Step 1.5: メンバー + Activities 統合（enrichment）
        ↓
Step 2: ルールベーススコアリング（5評価軸）
        ↓
Step 2.5: 潜在シナジー検知（スコア45-60点帯のみLLM）
        ↓
Step 3: ペア選定（スコアソート + 1人1件制限）
        ↓
Step 4: 紹介文生成（テンプレート版）
        ↓
Step 5: 未マッチング判定
        ↓
出力: マッチング結果 + 未マッチングメンバーリスト
     ↓ Notion DB に保存
```

## マッチング処理のフロー

### Step 0: マッチング済みペアの抽出

**目的**: 過去にマッチングしたペアの再マッチングを防止

```python
# マッチング履歴から frozenset([メンバーA, メンバーB]) を抽出
matched_pairs = { frozenset([中田誠治, 鈴鹿太郎]), ... }

# 過去に紹介した人数
matched_members = { 中田誠治, 鈴鹿太郎, ... }
```

**結果**: 過去にマッチングした同じペアの組み合わせは除外される

---

### Step 1: 全ペア生成と活動データ取得

**ペア生成アルゴリズム**

```python
# 全メンバー数: n 名の場合、ペア数 = n(n-1)/2
# 例: 10名 → 45組のペアを生成

import itertools
valid_pairs = []
for a, b in itertools.combinations(members, 2):  # すべてのペアを列挙
    pair_key = frozenset([a["名前"], b["名前"]])
    if pair_key not in matched_pairs:  # 既にマッチング済みなら除外
        valid_pairs.append((a, b))
```

**登録メンバー 10 名の場合の計算**

| ステップ | 件数 | 説明 |
|---------|------|------|
| 全ペア数 | 45 | 10 × 9 ÷ 2 |
| 既マッチペア | 0 | 初回なので 0 |
| 有効ペア数 | 45 | 45 - 0 |

### Step 1.5: Activities データの統合（Enrichment）

**マルチアクティビティ統合プロセス**

```python
def _merge_member_with_activities(member, activities):
    """
    基本情報 + 複数活動 → 統合プロフィール
    """
    merged = member.copy()
    
    # 複数活動から属性を抽出・統合
    industries = set()  # 対象業界を複数活動から集約
    strengths = []      # 強みをすべて統合
    issues = []         # 課題をすべて統合
    positions = []      # ポジションをすべて統合
    
    for activity in activities:
        industries.add(activity["対象業界"])
        strengths.extend(activity["強み"])  # 重複を許可
        issues.extend(activity["課題"])
        positions.extend(activity["ポジション"])
    
    # 重複排除
    merged["全対象業界"] = industries
    merged["全強み"] = list(set(strengths))
    merged["全課題"] = list(set(issues))
    merged["全ポジション"] = list(set(positions))
    merged["活動数"] = len(activities)
    
    return merged
```

## 5つのスコアリング評価軸

### 全体スコア計算式

```
Total Score = min(
    エンドクライアント一致度(0-20点)
    + バリューチェーン接続性(0-20点)
    + 市場ソリューションフィット(0-20点)
    + 事業拡張ポテンシャル(0-20点)
    + 対象市場一致度(0-20点)
    , 100
)

最小スコア(MIN_SCORE) = 45点 → これ以上のみマッチング対象
```

### 軸1️⃣: エンドクライアント一致度（0-20点）

**判定内容**: 同じクライアント層を狙っているか？互いに補完的か？

**計算ロジック**

| 判定項目 | 加算点 | 条件 |
|---------|-------|------|
| 対象業界の重複 | +4点 | A の対象業界 ∩ B の対象業界 |
| 業種カテゴリの一致 | +3点 | A.業種カテゴリ == B.業種カテゴリ |
| 対象企業規模の相性 | +2点 | A の対象規模 ∩ B の対象規模 |
| **Win-Win 双方向補完** | **+8点** | **A の強み が B の課題を解決 AND B の強み が A の課題を解決** |
| 片方向補完 | +3点 | A → B または B → A のみ |
| **上限** | **20点** | |

**具体例**

```
メンバーA（営業支援ツール企業）:
  強み: [営業支援, SaaS構築]
  課題: [営業開拓]
  対象業界: {IT}
  対象規模: {スタートアップ}

メンバーB（マーケティング代行企業）:
  強み: [マーケティング, リード生成]
  課題: [運用保守]
  対象業界: {IT}
  対象規模: {スタートアップ}

▼ 計算過程 ▼

1. 対象業界の重複: {IT} & {IT} = {IT} → ✓ +4点
2. 業種カテゴリの一致: SaaS == SaaS → ✓ +3点
3. 対象規模の相性: {スタ} & {スタ} → ✓ +2点
4. 双方向補完判定:
   - A の強み (営業支援) vs B の課題 (運用保守) → 重複なし
   - B の強み (マーケティング) vs A の課題 (営業開拓) → 重複あり ✓
   → 片方向のみ → +3点

合計: 4 + 3 + 2 + 3 = 12点
```

**同義語対応機能**

```python
# テキスト比較時に同義語グループを利用
KEYWORD_SYNONYMS = {
    "営業": ["営業", "セールス", "営業開発", "BDR"],
    "開発": ["開発", "エンジニアリング", "テック", "バックエンド"],
    "マーケティング": ["マーケティング", "マーケ", "ブランディング"],
    ...
}

# 例: "セールス" と "営業支援" を比較
# → "セールス" = "営業", "営業支援" = "営業" → 同じグループ → 重複判定 ✓
```

### 軸2️⃣: バリューチェーン接続性（0-20点）

**判定内容**: ポジション間に前後工程の関係があるか？

**相補的なポジション組み合わせ**

```python
complementary_pairs = [
    ("営業", "製造"),        # 営業 が売り、製造 が作る
    ("営業", "企画"),        # 営業 が売り、企画 が考える
    ("営業", "開発"),        # 営業 が売り、開発 が実装
    ("企画", "開発"),        # 企画 が設計、開発 が実装
    ("マーケティング", "営業"),  # マーケ が顧客を獲得、営業 が成約
    ("マーケティング", "開発"),  # マーケ が市場情報、開発 が新機能
    ("経営", "営業"),        # 経営 が戦略、営業 が実行
    ("経営", "開発"),        # 経営 が経営判断、開発 が実現
    ...
]
```

**計算ロジック**

| 判定項目 | 加算点 |
|---------|-------|
| 相補的ポジション組み合わせあり | +15点 |
| 異なるポジションセット | +5点 |
| **上限** | **20点** |

**具体例**

```
メンバーA:
  全ポジション: [営業, 企画]

メンバーB:
  全ポジション: [開発, デザイナー]

▼ 判定 ▼

1. 相補的ペアをチェック:
   - A の営業 & B の開発 → complementary_pairs に含まれる → 相補的 ✓
   → +15点

2. 異なるポジションセット:
   - {営業, 企画} != {開発, デザイナー} → ✓
   → +5点

合計: 15 + 5 = 20点（満点！）
```

**パターン分析**

| A のポジション | B のポジション | スコア | 理由 |
|--------------|--------------|--------|------|
| 営業 | 製造 | 20 | 前後工程で完全補完 |
| 企画 | 開発 | 20 | 設計と実装の補完 |
| 営業 | 営業 | 0 | 同じポジション、相補性なし |
| デザイナー | 営業 | 5 | 異なるが弱い補完 |

### 軸3️⃣: 市場ソリューションフィット（0-20点）

**判定内容**: 技術 × 営業、ソリューション × マーケットの相補性があるか？

**4つのパターン評価**

```python
def _calc_market_fit(a, b):
    # ★パターン1: 完全補完（一方が強み、他方が課題）
    if (a_has_solution and b_needs_solution and
        b_has_market and a_needs_market):
        score = 20
    
    # ★パターン2: ソリューション × マーケットの直接補完
    elif (a_has_solution and b_has_market) or \
         (b_has_solution and a_has_market):
        score = 15
    
    # ★パターン3: 同じ業界キーワード（新興）
    elif a_has_industry and b_has_industry:
        score = 12
    
    # ★パターン4: 部分的補完
    elif any_overlap:
        score = 8
    
    return min(score, 20)
```

**具体例**

```
メンバーA（技術系）:
  全強み: [Web開発, API設計, クラウド構築]
  全課題: [営業開拓]

メンバーB（営業系）:

▼ パターンマッチ ▼

パターン2: ソリューション (A) × マーケット (B) の直接補完
```

### 軸4️⃣: 事業拡張ポテンシャル（0-20点）

**判定内容**: 協業によって事業が広がる可能性があるか？

**計算ロジック**

| 判定項目 | 加算点 | 条件 |
|---------|-------|------|
| 業種の異なり | +12点 | A.業種カテゴリ ≠ B.業種カテゴリ |
| フェーズ相性（成長期×成長期） | +5点 | 両方とも成長期 |
| フェーズ相性（安定期×成長期） | +4点 | 一方が安定期、他方が成長期 |
| その他フェーズ組み合わせ | +2点 | 上記以外 |
| **上限** | **20点** | |

**具体例**

```
メンバーA:
  業種カテゴリ: IT
  事業フェーズ: 成長期

メンバーB:
  業種カテゴリ: コンサル
  事業フェーズ: 安定期

▼ 計算 ▼

1. 業種の異なり: IT ≠ コンサル → +12点
2. フェーズ相性: 成長期 × 安定期 → +4点

合計: 12 + 4 = 16点
→ 解釈: 異業種連携 + 安定期のリソースが成長期をサポート
```

### 軸5️⃣: 対象市場一致度（0-20点）

**判定内容**: 同じ顧客層（業界）を狙っているか？

**計算ロジック**

```python
# 対象業界の重複数に基づいてスコア計算
overlap_count = len(a_industries & b_industries)
score = min(overlap_count * 5, 20)

# 例:
# overlap_count = 1 → 1 × 5 = 5点
# overlap_count = 2 → 2 × 5 = 10点
# overlap_count = 3 → 3 × 5 = 15点（上限20点）
# overlap_count = 4+ → 20点（上限）
```

**具体例**

```
メンバーA:
  全対象業界: {金融, 医療, 不動産}

メンバーB:
  全対象業界: {金融, 製造, SaaS}

▼ 計算 ▼

1. 業界の重複: {金融, 医療, 不動産} & {金融, 製造, SaaS} = {金融}
2. 重複数: 1
3. スコア: 1 × 5 = 5点

→ 解釈: 金融市場での共同営業・サービス拡張の可能性
```

## 潜在シナジー検知メカニズム

### ハイブリッド判定システム（ルール + AI）

```
ルールベーススコア 45-60点帯
     ↓
属性マトリクス判定（4つのシグナル検知）
     ↓
シグナル数 ≥ 2 の場合のみ LLM 投入
     ↓
LLM で潜在シナジー を分析 → +0〜10点を加算
     ↓
最終スコア算出
```

### 4つのシナジーシグナル

```python
# Signal 1: フェーズ相乗効果
if (a_phase == "成長期" and b_phase == "安定期") or \
   (a_phase == "安定期" and b_phase == "成長期"):
    signals.append("フェーズ相乗")

# Signal 2: バリューチェーン補完（上流 × 下流）
if (a_is_upstream and b_is_downstream) or \
   (a_is_downstream and b_is_upstream):
    signals.append("バリューチェーン補完")

# Signal 3: 企業規模補完（SMB × 大企業）
if (a_is_smb and b_is_large) or \
   (a_is_large and b_is_smb):
    signals.append("企業規模補完")

# Signal 4: ドメイン知見の相補性（業界 × 機能）
# （より高度な属性判定）
```

### LLM 分析の例

```
入力（スコア 50点のペア）:

企業 A: 営業支援スタートアップ
- 強み: リード生成, 営業自動化
- 課題: 技術サポート不足

企業 B: データ分析コンサル（安定期）
- 強み: データ分析, BI構築
- 課題: 営業開拓困難

検出シグナル:
- フェーズ相乗（成長期 + 安定期）
- バリューチェーン補完（営業 × 分析）

↓ LLM 分析 ↓

出力:
{
  "有潜在シナジー": true,
  "加算点": 8,
  "理由": "成長期スタートアップの営業リソースと、安定期コンサルの分析スキルが統合される場合、
          データドリブン営業コンサルの新規事業創出可能"
}

最終スコア: 50 + 8 = 58点
```

### アルゴリズム

```python
# Step 5: マッチングされなかった人を抽出

matched_member_names = set()
for match in selected:
    matched_member_names.add(match["メンバーA名"])
    matched_member_names.add(match["メンバーB名"])

# selected に含まれなかった人をすべて抽出
unmatched_members = [m for m in members 
                    if m["名前"] not in matched_member_names]
```

### 未マッチングの理由パターン

| 理由 | 発生条件 | 例 |
|------|---------|-----|
| **スコア不足** | MIN_SCORE(45点) に未達 | スコア 30点のペアのみ → 全て 45点未満 |
| **相互補完性の欠落** | 強み・課題・ポジション が合致しない | 両者とも営業系のみ → 相補性なし |
| **選定漏れ** | スコア高いが 1人1件制限で除外 | スコア 60点のペアが 3つあり、メンバーが重複 |
| **業界不一致** | 対象業界が全く異なる | A: 医療、B: 金融 → overlap = {} |

## 技術的特徴

### 1. マルチアクティビティ対応

**従来版の問題**
```
問題: Members DB からのみ属性を取得
  → 複数活動の情報が失われていた
  → スコア max ~20点（品質不十分）

解決版:
  ✓ Activities DB から全活動を取得
  ✓ 複数活動の属性を union で統合
  ✓ スコア分布: 20-100点（品質向上）
```

**Enrichment パイプライン**
```
Member (名前, 会社, 業種) 
    ↓
get_activities_for_member() 
    ↓
Activities: [Activity1, Activity2, Activity3]
    ↓
_merge_member_with_activities()
    ↓
Enriched Member (全対象業界, 全強み, 全課題, etc.)
```

### 2. 同義語対応（テキスト偽陽性削減）

**問題**: 「Web開発」と「Webマーケティング」が「Web」で誤マッチ

**解決**:
```python
# 単語を同義語グループに正規化
"営業" group: [営業, セールス, 営業開発, BDR, ...]
"開発" group: [開発, エンジニアリング, バックエンド, ...]

# テキスト比較時に同グループを判定
"営業支援" の "営業" vs "営業開発" → 同じグループ → 重複 ✓
"Web開発" の "開発" vs "Webマーケティング" → 異なるグループ → 重複なし ✗
```

**効果**: テキスト偽陽性 30-40% 削減

### 3. 潜在シナジー検知（LLM の効率的活用）

**戦略**: 全ペアに LLM 使用ではなく、限定的投入

```
45-60点帯のペア（確信度中程度）のみに LLM 投入
  ↓
属性マトリクスで 2つ以上のシグナル検知時のみ実行
  ↓
LLM コスト削減 + 精度向上の両立
```

**コスト試算**
```
ペア数: 45組
LLM 投入候補: 45-60点帯 = 平均 5-10組
実際に LLM 実行: シグナル ≥ 2 の場合のみ = 平均 2-3組
削減率: ~90%

API コスト: 45組中 3組 × ~$0.0001 = ~$0.0003/回
```

### 4. 再マッチング防止メカニズム

**データ構造**
```python
matched_pairs = {
    frozenset([名前A, 名前B]),  # ペアを frozenset で管理
    frozenset([名前C, 名前D]),
    ...
}
```

**特徴**
```
✓ フローリングセットで順序不変
  → (中田誠治, 加藤由美) == (加藤由美, 中田誠治)

✓ 高速な O(1) ルックアップ
  → 45ペアの確認が瞬間

✓ 複数回実行時の再利用可能
  → Notion DB の MATCHING_HISTORY から自動復元
```

### 5. スコア再現性確保

**同一スコアのソート**
```python
# スコアが同じ場合は、メンバーA名の辞書順でソート
scored.sort(key=lambda x: (-x["スコア"], x["メンバーA名"], x["メンバーB名"]))

# 効果:
# - 複数回実行しても同じ結果
# - バッチ処理時の一貫性確保
```
