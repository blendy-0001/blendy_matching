# フォーム再設計：マルチアクティビティ対応

## 現状の問題点

### ❌ 単一アクティビティモデルの限界

**例：人材紹介会社**
- 人材紹介業 (営業・提案・クロージング) → 採用企業を支援
- 交流会・マッチングイベント (認知・ブランディング) → フリーランスを支援

**現在のフォーム:**
- 「主力サービス」= 1つだけ
- 「バリューチェーン位置」= 会社全体で 1 セット
- 「対象クライアント業界/規模」= 1つだけ

→ この企業の「複数の価値チェーンポジション」を同時に記録できない
→ マッチング精度が低下（異なる活動に対応した相互補完パートナーを見落とす）

---

## 改善設計：マルチアクティビティモデル

### 📊 新フォーム構造

```
フォーム全体
├─ 基本情報（会社共通）
│  ├─ 名前
│  ├─ 会社名
│  ├─ 業種カテゴリ（大分類）
│  ├─ 業種詳細
│  └─ 事業フェーズ
│
├─ 事業アクティビティ（複数登録可能）
│  ├─ [Activity 1]
│  │  ├─ アクティビティ名
│  │  ├─ サービス・事業内容（詳細説明）
│  │  ├─ 強み（このアクティビティでの強み）
│  │  ├─ 課題（このアクティビティでの課題）
│  │  ├─ バリューチェーン位置（このアクティビティでの立場）
│  │  ├─ 対象クライアント業界
│  │  └─ 対象クライアント規模
│  │
│  ├─ [Activity 2]
│  │  └─ （同じ構成...）
│  │
│  └─ [Activity 3] (オプション)
│     └─ （同じ構成...）
│
└─ 連絡先・その他（会社共通）
   ├─ メール
   ├─ LINE ID
   └─ Facebook URL
```

### 🎯 主な改善点

#### 1. アクティビティごとの個別情報

| 項目 | 現在 | 改善後 |
|------|------|--------|
| 主力サービス | テキスト 1個 | アクティビティ × N個 |
| バリューチェーン位置 | 会社共通 1セット | アクティビティごとに個別 |
| 対象クライアント | 会社共通 1セット | アクティビティごとに個別 |
| 強み・課題 | 会社全体で 1セット | アクティビティごとに個別 |

#### 2. ユースケース別の記載例

**例 1：人材紹介会社**

| フィールド | Activity 1: 人材紹介 | Activity 2: 交流会主催 |
|-----------|------------------|-------------------|
| サービス内容 | 採用企業向けエグゼクティブ人材紹介（年間売上3億）| フリーランス/起業家向けマッチングイベント月1回開催 |
| 強み | 業界ネットワーク10年分・大手企業との取引実績 | イベント企画力・参加者との信頼構築 |
| 課題 | 営業マーケティング力不足（新規営業できない） | 認知度拡大・参加者の質向上 |
| バリューチェーン位置 | 営業・提案・クロージング | 認知・ブランディング |
| 対象クライアント | 採用企業（300名以上）| フリーランス・スモールビジネス |

→ この企業は「営業支援パートナー（A型）」と「マーケティング/PR企業（C型）」の両方とマッチ可能に

**例 2：コンサルティング会社（複数業界展開）**

| フィールド | Activity 1: IT企業向け | Activity 2: 製造業向け |
|-----------|------------------|-------------------|
| サービス内容 | DX推進コンサルティング | 業務効率化・生産性向上コンサル |
| 強み | IT業界知見・デジタル技術 | 製造業特有の課題解決実績 |
| 課題 | 営業・初期営業接点がない | 技術開発（実装支援）がない |
| バリューチェーン位置 | 営業・提案・クロージング + リード獲得 | 営業・提案・クロージング + コンサル |
| 対象クライアント | IT企業（従業員10〜300名）| 製造業（従業員300名以上）|

### スコアリングロジック変更

#### 現在

```python
def score_pair(member_a, member_b):
    # member_a: 1つのバリューチェーン位置
    # member_b: 1つのバリューチェーン位置
    score = calculate_complementarity(member_a.positions, member_b.positions)
    return score
```

#### 改善後

```python
def score_pair_multiactivity(member_a, member_b):
    """複数のアクティビティを持つメンバー間のマッチングスコア"""
    scores = []
    
    # member_a の各アクティビティ × member_b の各アクティビティで全組み合わせをスコア
    for activity_a in member_a.activities:
        for activity_b in member_b.activities:
            activity_score = calculate_activity_complementarity(
                activity_a, activity_b
            )
            if activity_score > MIN_SCORE:
                scores.append({
                    'activity_pair': (activity_a.name, activity_b.name),
                    'score': activity_score,
                    'type': activity_score['type']  # A型/B型/C型など
                })
    
    # ベストマッチ（最高スコア＋複数の相互補完パターン）
    if scores:
        best = max(scores, key=lambda x: x['score']['total'])
        return {
            'overall_score': best['score']['total'],
            'primary_match': best['activity_pair'],
            'match_type': best['type'],
            'all_matches': scores  # マッチする活動ペアの全リスト
        }
    return None
```

## マイグレーション戦略

### Phase 1: フォーム移行（互換性維持）

**初期状態：**
- 新フォームを `register_new.html` として展開
- 既存ユーザーは旧フォーム（単一活動）で記載済み

**移行方針：**
1. 既存ユーザーの「主力サービス」「バリューチェーン」を Activity 1 に自動変換
2. 既存ユーザーに「追加活動がありますか？」を LINE/Email で案内
3. 新規ユーザーは新フォームで複数活動を最初から記載

### Phase 2: スコアリング更新

```python
# 既存ユーザー（単一活動）
def backward_compatible_score(member_a, member_b):
    # Activity が1つの場合は旧ロジックを使用
    if len(member_a.activities) == 1 and len(member_b.activities) == 1:
        return old_scoring_logic(member_a, member_b)
    
    # 複数活動がある場合は新ロジック
    return score_pair_multiactivity(member_a, member_b)
```

## 期待される改善効果

### 📈 マッチング精度向上

| 指標 | 現状 | 改善後 |
|------|------|--------|
| 記録可能な活動数 | 1個 | 3個 |
| 活動ごとの補完性検出 | 不可 | 可能 |
| 複数相互補完パターン検出 | なし | A型+C型など複数パターン |
| 対等性判定の精度 | 会社単位（粗） | 活動単位（細） |

### 🎯 具体的な改善ケース

**ケース 1：人材紹介会社 vs. マーケティング会社**

- **現在：** 「営業」と「マーケティング」で部分的に相互補完 → スコア 55〜65
- **改善後：** 
  - Activity 1: 人材紹介（営業・提案）✓ 補完可能 → スコア 75
  - Activity 2: 交流会（ブランディング）✓ PR企業が補完 → スコア 68
  - 複合スコア: 72（平均） → より高精度な推奨

**ケース 2：コンサルがIT企業と製造業の両方に営業中**

- **現在：** 「汎用コンサル」として大雑把に分類 → スコア曖昧
- **改善後：**
  - Activity 1: IT向けDXコンサル → IT企業のDX支援企業と相互補完
  - Activity 2: 製造業向け業務効率化 → 製造業特化の営業パートナーと相互補完
  - 業界別の最適パートナーをそれぞれ発見
